# AI 学习打卡助手第一版 MVP 产品需求文档

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | 知知小助手 |
| 文档类型 | 产品需求文档 PRD |
| 版本 | V1.0 / MVP 版本 |
| 目标用户 | 初中学生家庭，优先面向家长与孩子双端使用 |
| 核心目标 | 通过“家长布置任务 → 孩子打卡上传 → AI 检查 → 家长确认发放积分 → 孩子兑换心愿 → 周报复盘”的闭环，帮助家长降低检查成本，帮助孩子建立学习执行习惯，并逐步发现薄弱点 |

---

## 2. 产品背景

### 2.1 业务背景

小孩即将从小学升入民办初中，学习节奏、作业量、考试频率和竞争压力都会明显提升。家长希望提前帮助孩子建立稳定的学习执行机制，同时能够持续掌握孩子每天的学习完成情况。

当前家庭学习管理中存在几个典型问题：

1. 家长每天布置了学习任务，但无法实时确认孩子是否真正完成。
2. 孩子完成任务后，家长仍需要花大量时间逐项检查。
3. 很多问题只停留在“做完没做完”，缺少对错题、薄弱知识点和学习状态的持续追踪。
4. 家长容易焦虑，但缺少结构化数据判断孩子是否在进步。
5. 孩子也容易对学习安排缺少清晰感，不知道每天具体要完成什么。

### 2.2 产品机会

产品可以在家庭学习管理中承担四类价值：

1. **过程记录**：让孩子每天完成任务后勾选并拍照上传，形成可追踪记录。
2. **辅助检查**：AI 初步判断上传内容是否与任务匹配、是否完整、是否存在明显错误或敷衍。
3. **复盘分析**：基于一周数据生成学习报告，帮助家长发现执行问题和薄弱点，并指导下周计划。
4. **正向激励**：通过家长设置任务积分、孩子提交愿望、积分兑换心愿，把学习完成反馈从单纯监督扩展为可持续的家庭激励机制。

第一版 MVP 的重点不是做复杂的智能教学系统，而是验证一个家庭学习管理闭环是否成立。

## 3. 产品定位

### 3.1 一句话定位

面向家长和孩子的 AI 学习任务打卡工具，帮助家长布置每日学习计划、监督孩子完成情况，并通过 AI 降低检查成本、发现薄弱点。

### 3.2 产品核心价值

| 用户 | 核心价值 |
|---|---|
| 家长 | 每天清楚知道孩子完成了什么，减少人工检查时间，获得每周查漏补缺建议 |
| 孩子 | 每天任务更清晰，完成后有反馈和积分激励，有助于养成学习执行习惯 |
| 家庭 | 从“家长口头催促”变成“任务清单 + 打卡记录 + AI 反馈 + 积分心愿 + 周报复盘”的数据化闭环 |

### 3.3 产品不做什么

MVP 阶段暂不做以下内容：

1. 不做完整在线教学平台。
2. 不做复杂课程体系。
3. 不做直播课、录播课、班级管理。
4. 不做多孩子、多班级、多老师复杂组织架构。
5. 不追求 AI 自动批改所有题型的高准确率。
6. 不在第一版强行做完整个性化学习路径推荐。

第一版只验证：任务是否能布置清楚、孩子是否愿意打卡、上传是否真实有效、AI 是否能减少家长检查成本、周报是否能指导下周行动。

## 4. 目标用户与用户角色

## 4.1 家长

### 用户特征

1. 关注孩子小升初、民办初中适应问题。
2. 希望孩子养成每日学习习惯。
3. 对孩子学习进度有焦虑，但没有足够时间每天细查。
4. 希望通过工具获得清晰、可视化、可复盘的数据。

### 核心诉求

1. 每天能方便地布置任务。
2. 能看到孩子是否完成、何时完成、上传了什么。
3. AI 能帮忙初步检查完成度，减少重复检查。
4. 每周知道孩子哪些科目或知识点比较薄弱。
5. 能据此调整下一周学习计划。

## 4.2 孩子

### 用户特征

1. 小学高年级或初中预备阶段。
2. 自律能力尚未完全建立。
3. 对复杂系统和繁琐操作敏感。
4. 需要明确、简单、可完成的任务清单。

### 核心诉求

1. 每天知道自己需要完成什么。
2. 操作要简单，不能像写报告一样麻烦。
3. 完成后能快速打卡。
4. 希望获得正向反馈，而不是只有批评。

## 4.3 AI 助手

AI 在产品中不是独立用户，但承担辅助角色：

1. 检查上传内容是否匹配任务。
2. 判断任务完成度。
3. 提取错题和薄弱点。
4. 汇总每日、每周学习表现。
5. 给家长提供可执行建议。

## 5. 产品目标

### 5.1 MVP 阶段目标

第一版 MVP 需要验证以下核心假设：

1. 家长愿意每天在系统中创建学习任务。
2. 孩子愿意查看任务并完成打卡。
3. 拍照上传可以作为有效的学习完成凭证。
4. AI 检查能够明显减少家长的人工检查成本。
5. 每周报告能帮助家长调整下一周学习计划。
6. 愿望清单和积分兑换能提升孩子持续完成任务的积极性。

### 5.2 核心业务指标

| 指标 | 说明 | MVP 目标 |
|---|---|---|
| 家长任务创建率 | 家长每天是否创建任务 | 连续 7 天内创建任务天数 ≥ 5 天 |
| 孩子打卡完成率 | 孩子完成并提交任务的比例 | 每日任务完成率 ≥ 70% |
| 上传有效率 | 上传图片是否能被家长/AI 判断为有效凭证 | 有效上传比例 ≥ 80% |
| AI 检查可用率 | AI 检查结果被家长认为有参考价值的比例 | ≥ 60% |
| 周报阅读率 | 家长是否查看每周报告 | ≥ 70% |
| 家长复用率 | 家长第二周是否继续使用 | ≥ 50% |
| 愿望激励使用率 | 孩子是否提交愿望，家长是否设置积分并完成兑换闭环 | 至少完成 1 次愿望提交和 1 次积分兑换申请 |

## 6. 核心使用流程

## 6.1 家长侧主流程

1. 家长进入系统。
2. 创建今日学习任务。
3. 选择科目、任务类型、任务内容、预计完成时间。
4. 保存并发布给孩子。
5. 孩子完成任务后，家长在今日看板查看完成状态。
6. 家长查看孩子上传的图片和 AI 检查结果。
7. 家长根据情况确认、驳回或备注。
8. 家长确认任务完成后，系统按任务积分发放给孩子。
9. 家长查看孩子提交的愿望，设置所需积分或驳回。
10. 孩子发起心愿兑换申请后，家长确认兑换并扣减积分。
11. 每周查看 AI 生成的学习报告。
12. 根据周报建议调整下周计划。

## 6.2 孩子侧主流程

1. 孩子进入系统。
2. 查看今日任务清单。
3. 完成某项学习任务。
4. 勾选完成。
5. 拍照上传作业、练习册、错题或其他凭证。
6. 提交打卡。
7. 查看提交成功状态和简单反馈。
8. 家长确认任务完成后获得对应积分。
9. 提交自己的愿望。
10. 积分足够时发起心愿兑换申请。
11. 如任务被家长驳回，则重新补充上传。

## 6.3 AI 检查流程

1. 孩子上传图片。
2. 系统识别任务信息与图片内容。
3. AI 判断图片是否与任务相关。
4. AI 判断完成度。
5. AI 提取可能的错题、空题、未完成区域或明显问题。
6. AI 生成简短检查结论。
7. 结果展示给家长。

## 6.4 每周报告流程

1. 系统汇总一周任务数据。
2. 统计每日完成情况、科目分布、任务类型分布。
3. 统计未完成、被驳回、AI 识别异常的任务。
4. 汇总错题和薄弱点。
5. 生成周报。
6. 给出下周建议。
7. 家长查看并可一键生成下周任务草稿。

## 6.5 愿望清单与积分兑换流程

1. 家长创建任务时设置该任务可获得的积分。
2. 孩子完成任务并提交打卡。
3. 家长确认任务完成。
4. 系统为孩子发放该任务积分，并记录积分流水。
5. 孩子提交愿望名称和愿望说明。
6. 家长查看愿望，只能设置所需积分并通过，或直接驳回，不能修改孩子提交的愿望原文。
7. 愿望通过后进入可兑换状态。
8. 孩子积分达到愿望所需积分后发起兑换申请。
9. 家长确认兑换。
10. 系统扣减积分，记录积分流水，并将愿望标记为已兑换。

## 7. 功能范围

## 7.1 第一版最关键的 8 个功能

1. 家长创建每日任务。
2. 孩子查看任务清单。
3. 孩子勾选完成并拍照上传。
4. AI 检查上传内容和完成度。
5. 家长查看今日完成看板。
6. 愿望清单和积分兑换。
7. AI 生成每周学习报告。
8. 日历面板按日期查看、创建、编辑和删除任务。

## 7.2 优先级排序

| 优先级 | 模块 | 说明 |
|---|---|---|
| P0 | 任务打卡闭环 | MVP 成立的基础，必须优先完成 |

| P1 | 积分与愿望清单 | 提升孩子持续完成任务的正向反馈，依赖家长确认任务完成 |
| P1 | AI 完成度检查 | 降低家长检查成本的核心价值 |
| P1 | 错题记录 | 为后续薄弱点分析提供数据基础 |
| P2 | 薄弱点分析 | 从任务完成升级到学习诊断 |
| P2 | 周报生成 | 帮助家长复盘并指导下周行动 |

# 8. 功能需求详细说明

## 8.1 家长创建每日任务

### 功能目标

帮助家长快速创建清晰、可执行、可检查的每日学习任务。

### 使用对象

家长。

### 前置条件

1. 家长已使用用户名和密码登录。
2. 家长已绑定孩子账号。

### 主要字段

| 字段 | 说明 | 是否必填 |
|---|---|---|
| 任务标题 | 如“完成数学计算练习第 3 页” | 是 |
| 科目 | 语文、数学、英语、其他 | 是 |
| 任务类型 | 作业、预习、复习、错题、阅读、背诵、练习 | 是 |
| 任务说明 | 详细要求，如页码、题号、完成标准 | 是 |
| 截止时间 | 当天几点前完成 | 否 |
| 任务积分 | 家长设置，孩子完成并经家长确认后获得，非负整数，默认 1 分 | 是 |
| 是否需要拍照 | 默认需要 | 是 |
| 是否需要 AI 检查 | 默认开启 | 否 |
| 备注 | 给孩子看的补充说明 | 否 |

### 操作流程

1. 家长点击“创建任务”。
2. 选择孩子。
3. 填写任务信息。
4. 点击“保存并发布”。
5. 任务进入孩子今日任务清单。

### 验收标准

1. 家长可以在 1 分钟内创建一条任务。
2. 任务保存后孩子端可立即查看。
3. 任务至少包含标题、科目、任务类型、任务说明。
4. 任务创建时必须有任务积分，默认 1 分，家长可调整为 0 分。
5. 任务创建后状态默认为“待完成”。
6. 家长可以编辑未完成任务，包括任务积分。
7. 家长可以删除未完成任务。

---

## 8.2 孩子查看任务清单

### 功能目标

让孩子清楚知道今天需要完成哪些任务，以及每项任务的完成要求。

### 使用对象

孩子。

### 页面内容

1. 今日任务列表。
2. 每项任务展示科目、标题、任务说明、状态。
3. 状态包括：待完成、已提交、AI 检查中、家长已确认、需补充。
4. 支持按状态筛选。
5. 支持查看任务详情。

### 操作流程

1. 孩子打开今日任务。
2. 查看任务列表。
3. 点击某条任务查看详情。
4. 完成任务后进入打卡提交。

### 验收标准

1. 孩子能看到当天所有任务。
2. 已完成与未完成任务有明显区分。
3. 每项任务说明要清晰展示。
4. 操作路径不超过 3 步进入打卡。

---

## 8.3 孩子勾选完成并拍照上传

### 功能目标

让孩子以最低成本提交任务完成凭证。

### 使用对象

孩子。

### 功能说明

孩子完成任务后，可以勾选完成并上传照片作为凭证。照片可以是作业本、练习册、试卷、错题本、阅读笔记、背诵打卡页面等。

### 主要字段

| 字段 | 说明 |
|---|---|
| 任务 ID | 当前打卡对应的任务 |
| 完成状态 | 孩子主动勾选完成 |
| 上传图片 | 支持 1 到 9 张 |
| 孩子备注 | 可选，如“第 5 题不会” |
| 提交时间 | 系统自动记录 |

### 操作流程

1. 孩子点击任务。
2. 点击“完成并打卡”。
3. 勾选“我已完成”。
4. 拍照或从相册上传。
5. 可填写备注。
6. 点击提交。
7. 任务状态变更为“已提交 / AI 检查中”。

### 图片上传要求

1. 支持拍照和相册选择。
2. 单个任务最多上传 9 张图片。
3. 图片需压缩后上传。
4. 上传失败时支持重新上传。
5. 图片需要记录上传时间。

### 验收标准

1. 孩子可以完成打卡提交。
2. 家长端可以看到上传图片。
3. 图片上传失败时有明确提示。
4. 已提交任务不可重复提交，但可在“需补充”状态下再次上传。
5. 提交后触发 AI 检查任务。

---

## 8.4 AI 检查上传内容和完成度

### 功能目标

通过 AI 初步判断孩子上传内容是否与任务匹配、是否完成、是否存在明显问题，从而减少家长人工检查成本。

### 使用对象

家长、系统 AI。

### AI 检查输入

1. 任务标题。
2. 任务科目。
3. 任务说明。
4. 孩子上传图片。
5. 孩子备注。
6. 历史错题记录，可选。

### AI 检查输出

| 输出项 | 说明 |
|---|---|
| 匹配度 | 上传内容是否与任务要求相关 |
| 完成度 | 已完成、部分完成、疑似未完成、无法判断 |
| 异常提示 | 图片模糊、内容不完整、页码不符、空题较多等 |
| 错题提取 | 识别可能做错或孩子标注不会的题目 |
| 简短结论 | 给家长看的自然语言总结 |
| 建议动作 | 确认通过、建议复查、要求补充上传 |

### AI 检查状态

| 状态 | 说明 |
|---|---|
| 待检查 | 孩子已提交，等待 AI 处理 |
| 检查中 | AI 正在识别图片和生成结果 |
| 检查完成 | 已生成检查结果 |
| 检查失败 | 图片或模型处理失败 |
| 需人工复查 | AI 置信度较低，需要家长确认 |

### AI 判断规则

#### 8.4.1 任务匹配判断

AI 需要判断上传内容是否和任务要求一致。

示例：

| 任务 | 上传内容 | 判断 |
|---|---|---|
| 完成数学练习册第 5 页 | 上传数学练习册第 5 页 | 匹配 |
| 完成英语单词抄写 | 上传语文作文 | 不匹配 |
| 完成阅读笔记 | 上传一张空白纸 | 疑似无效 |

#### 8.4.2 完成度判断

AI 按以下等级输出：

| 等级 | 说明 |
|---|---|
| 完成 | 图片中任务内容基本完成 |
| 部分完成 | 有明显空题、缺页、只完成一部分 |
| 疑似未完成 | 上传内容过少、空白、与任务不符 |
| 无法判断 | 图片模糊、遮挡、角度不佳 |

#### 8.4.3 异常识别

AI 需要尽量识别以下异常：

1. 图片模糊。
2. 图片角度严重倾斜。
3. 内容被遮挡。
4. 上传页码疑似不符。
5. 题目大量空白。
6. 与任务科目不匹配。
7. 只上传封面，没有上传实际完成内容。

### 家长确认机制

AI 结果只作为辅助，不直接替代家长最终判断。

家长可以对 AI 检查结果进行以下操作：

1. 确认通过。
2. 要求孩子补充上传。
3. 标记 AI 判断不准确。
4. 添加家长备注。

### 验收标准

1. 孩子上传后系统自动触发 AI 检查。
2. 家长能看到 AI 检查结果。
3. AI 至少输出完成度、异常提示和建议动作。
4. AI 无法判断时必须明确提示，而不是强行给结论。
5. 家长可以覆盖 AI 判断结果。
6. AI 检查失败时不影响家长查看原图。

## 8.5 家长查看今日完成看板

### 功能目标

让家长快速掌握孩子当天学习任务完成情况。

### 使用对象

家长。

### 页面内容

今日看板需要展示：

1. 今日任务总数。
2. 已完成数量。
3. 未完成数量。
4. 需补充数量。
5. AI 提示异常数量。
6. 按科目展示完成情况。
7. 每个任务的状态、任务积分、上传图片、AI 检查结论。
8. 孩子当前可用积分入口。

### 今日任务状态

| 状态 | 说明 |
|---|---|
| 待完成 | 孩子尚未提交 |
| 已提交 | 孩子已打卡，等待 AI 或家长确认 |
| AI 检查中 | 系统正在处理 |
| 待家长确认 | AI 已检查完成，等待家长确认 |
| 已确认 | 家长确认完成 |
| 需补充 | 家长要求孩子重新补充 |

### 操作流程

1. 家长进入今日看板。
2. 查看整体完成率。
3. 点击异常任务。
4. 查看图片和 AI 检查结论。
5. 选择确认通过或要求补充。
6. 确认通过后系统发放该任务积分。
7. 必要时添加备注。

### 验收标准

1. 家长打开页面即可看到今日整体完成情况。
2. 异常任务需要有明显标记。
3. 家长可以快速查看每项任务的图片和 AI 结论。
4. 家长可以确认任务或要求补充。
5. 家长确认任务完成后，任务积分只能发放一次。
6. 数据状态实时更新。

## 8.6 愿望清单与积分兑换

### 功能目标

通过家庭内部的积分和愿望兑换机制，给孩子建立清晰、可预期的正向反馈，提升持续完成学习任务的动力。

### 使用对象

家长、孩子。

### 功能说明

孩子可以提交自己的愿望。家长查看后，只能设置愿望所需积分并通过，或直接驳回，不能修改孩子提交的愿望名称和说明。孩子通过完成任务获得积分，积分仅在家长确认任务完成后发放。孩子积分足够时，可以发起兑换申请，家长确认兑换后系统扣减积分并标记愿望已兑换。

### 业务规则

1. 任务积分由家长在创建或编辑未完成任务时设置。
2. 任务积分为非负整数，默认 1 分，可设置为 0 分。
3. 孩子提交任务、AI 判断完成或任务进入待家长确认状态时，均不直接发放积分。
4. 只有家长确认任务完成后，系统才发放该任务积分。
5. 同一任务积分只能发放一次，避免重复确认导致重复加分。
6. 孩子提交愿望后，愿望进入“待家长设置积分”状态。
7. 家长可以为愿望设置所需积分并通过，也可以驳回愿望。
8. 家长不能修改孩子提交的愿望名称和说明。
9. 孩子当前积分不足时不能发起兑换申请。
10. 孩子积分足够时，可以发起兑换申请。
11. 家长确认兑换后，系统扣减对应积分，并将愿望标记为“已兑换”。
12. 系统必须记录积分获得和扣减流水，方便追踪积分来源和去向。
13. 第一版周报不展示积分和愿望数据。
14. 被家长驳回的愿望，孩子可以在孩子端重新编辑或删除。编辑保存后愿望回到“待家长设置积分”状态，原所需积分、家长审核记录和驳回原因会被清空。

### 愿望主要字段

| 字段 | 说明 | 是否必填 |
|---|---|---|
| 愿望标题 | 孩子填写，如“买一本漫画书”“周末去科技馆” | 是 |
| 愿望说明 | 孩子填写，可描述具体心愿 | 否 |
| 所需积分 | 家长设置，非负整数 | 家长审核时必填 |
| 愿望状态 | 待家长设置积分、可兑换、已驳回、兑换申请中、已兑换 | 是 |
| 家长备注 | 家长驳回或兑换确认时可填写 | 否 |
| 创建时间 | 系统记录 | 是 |

### 积分流水字段

| 字段 | 说明 |
|---|---|
| 孩子 ID | 积分所属孩子 |
| 变动积分 | 正数表示获得，负数表示扣减 |
| 变动原因 | 任务完成奖励、心愿兑换扣减 |
| 来源类型 | task_review / wish |
| 来源 ID | 关联家长确认记录或愿望 ID |
| 操作人 | 系统或家长用户 |
| 创建时间 | 系统记录 |

### 孩子操作流程

1. 孩子进入愿望清单。
2. 查看当前可用积分。
3. 点击“提交愿望”。
4. 填写愿望标题和说明。
5. 提交后等待家长设置积分或驳回。
6. 愿望通过且积分足够时，点击“申请兑换”。
7. 等待家长确认兑换。
8. 愿望被家长驳回时，可点击“修改心愿”重新编辑后再次提交，或点击“删除心愿”直接删除该心愿。

### 家长操作流程

1. 家长进入愿望管理。
2. 查看孩子提交的愿望。
3. 对待处理愿望设置所需积分并通过，或驳回。
4. 查看孩子发起的兑换申请。
5. 确认兑换后，系统扣减积分并标记愿望已兑换。

### 愿望状态

| 状态 | 说明 |
|---|---|
| 待家长设置积分 | 孩子已提交，等待家长处理 |
| 可兑换 | 家长已设置所需积分，孩子可积累积分兑换 |
| 已驳回 | 家长驳回该愿望 |
| 兑换申请中 | 孩子积分足够并已发起兑换申请 |
| 已兑换 | 家长确认兑换完成，系统已扣减积分 |

### 验收标准

1. 孩子可以提交愿望标题和说明。
2. 家长可以查看孩子提交的愿望。
3. 家长只能设置所需积分并通过，或驳回愿望，不能修改孩子愿望原文。
4. 家长创建任务时可以设置任务积分。
5. 家长确认任务完成后，系统自动给孩子发放该任务积分。
6. 同一任务不会重复发放积分。
7. 孩子可以查看当前可用积分和愿望状态。
8. 孩子积分足够时可以发起兑换申请。
9. 家长确认兑换后系统扣减积分，并将愿望标记为已兑换。
10. 积分获得和扣减必须有流水记录。
11. 周报不展示积分和愿望数据。
12. 孩子可以修改被家长驳回的愿望，保存后状态回到“待家长设置积分”。
13. 孩子可以删除被家长驳回的愿望，删除后该记录不再出现。
6. 同一任务不会重复发放积分。
7. 孩子可以查看当前可用积分和愿望状态。
8. 孩子积分足够时可以发起兑换申请。
9. 家长确认兑换后系统扣减积分，并将愿望标记为已兑换。
10. 积分获得和扣减必须有流水记录。
11. 周报不展示积分和愿望数据。

## 8.7 日历面板

### 功能目标

日历面板参考 Apple 日历月视图的清晰、轻量、按日期聚合的展示方式，让任务不只以列表形式出现，也能按日期规划和管理。

核心目标：

1. 让家长和孩子可以通过日历视图直观看到每天有哪些任务。
2. 让任务不只是列表展示，也能按日期规划和管理。
3. 降低家长安排任务的操作成本。
4. 提升孩子对每日任务节奏的感知。

### 使用对象

家长、孩子。

### 用户角色与权限

#### 家长

1. 可以进入日历面板查看当前家庭内所有孩子的任务。
2. 可以在日历中创建任务。
3. 可以在日历中编辑任务。
4. 可以在日历中删除未完成任务。
5. 可以调整任务日期、任务积分、任务说明、完成要求、所属孩子、截止时间、是否需要照片证明。
6. 已完成任务原则上不建议随意修改积分；如果必须修改，需要二次确认，并明确提示会影响积分结算和积分流水一致性。
7. 已提交审核中的任务修改时，需要提示家长可能影响孩子已提交内容和审核判断。

#### 孩子

1. 可以进入日历面板查看分配给自己的任务。
2. 可以查看任务日期、任务标题、任务内容、任务积分和任务状态。
3. MVP 阶段孩子不允许创建任务。
4. MVP 阶段孩子不允许编辑或删除任务。
5. 是否允许孩子自主创建任务作为后续扩展，不纳入 MVP。

### 日历视图要求

1. 默认展示月视图。
2. 日历顶部显示当前年月。
3. 支持切换上个月 / 下个月。
4. 支持回到今天。
5. 每个日期格子中展示当天任务。
6. 今天需要有明显标识。
7. 有任务的日期需要显示任务摘要。
8. 支持点击某一天创建新任务。
9. 支持点击已有任务查看详情。
10. 支持移动端基础适配。

### 任务展示规则

每个日历格子内的任务至少展示：

1. 任务标题。
2. 任务状态。
3. 任务积分。
4. 所属孩子。
5. 是否已完成 / 待审核 / 已审核。

建议展示状态：

| 展示状态 | 对应含义 |
|---|---|
| 待完成 | 孩子尚未提交 |
| 已提交 | 孩子已提交打卡 |
| 待家长审核 | AI 检查完成或任务进入家长确认环节 |
| 已完成 | 家长已确认完成 |
| 已驳回 | 家长要求补充或任务被判定需要重新提交 |
| 已过期 | 任务日期或截止时间已过，但任务仍未完成 |

状态映射优先复用现有任务状态，不新建一套日历专用状态。日历中的状态展示必须与任务列表、任务详情保持一致。

### 创建任务

家长在日历面板中点击某个日期后，可以创建任务。创建任务默认日期为当前点击的日期。

创建任务字段包括：

| 字段 | 说明 | 是否必填 |
|---|---|---|
| 任务标题 | 如“完成数学计算练习第 3 页” | 是 |
| 任务说明 | 给孩子看的详细说明 | 是 |
| 任务日期 | 默认取点击的日历日期，可调整 | 是 |
| 所属孩子 | 从当前家庭孩子中选择 | 是 |
| 任务积分 | 家长设置，家长确认完成后发放 | 是 |
| 完成要求 | 如页码、题号、完成标准、背诵要求等 | 是 |
| 是否需要上传照片证明 | 默认需要 | 是 |
| 截止时间 | 可选，如当天 20:30 前 | 否 |
| 是否重复任务 | MVP 阶段不做，仅作为后续扩展 | 否 |

### 编辑任务

家长可以在日历中点击任务后进入任务详情，并从详情页编辑任务。

可编辑字段：

1. 标题。
2. 说明。
3. 日期。
4. 积分。
5. 所属孩子。
6. 完成要求。
7. 截止时间。
8. 是否需要照片证明。

编辑规则：

1. 未完成任务可以编辑上述字段。
2. 已完成任务原则上不建议修改积分。
3. 如果必须修改已完成任务积分，需要二次确认，并提示“修改积分可能影响已发放积分和积分流水一致性”。
4. 已提交审核中的任务修改时，需要提示“修改任务要求可能影响孩子已提交内容，请确认是否继续”。
5. 编辑保存成功后，日历中的任务信息需要立即刷新。

### 删除任务

家长可以删除未完成任务。

删除规则：

1. 未完成任务可以删除。
2. 已完成并已发放积分的任务，不建议硬删除。
3. 已完成任务应采用“归档 / 作废”方式处理，避免积分记录不一致。
4. 删除前必须二次确认。
5. 删除成功后，任务不再出现在日历面板和任务列表中。

### 与现有任务系统的关系

1. 日历面板不是一套新的任务系统，而是现有任务数据的日历化展示与操作入口。
2. 从任务列表创建的任务，也应出现在日历中。
3. 从日历创建的任务，也应出现在任务列表中。
4. 日历、任务列表、任务详情的数据必须一致。
5. 日历创建、编辑、删除任务时，仍然使用现有任务的权限、状态流转和家庭数据隔离规则。

### 与积分系统的关系

1. 日历创建的任务仍然遵循现有积分规则。
2. 孩子完成任务后，家长审核通过才发放积分。
3. 任务日期用于规划，不等于自动发放积分。
4. AI 判断完成、孩子提交打卡、任务进入待审核状态，都不直接发放积分。
5. 过期任务是否可补交，先按照现有任务规则处理。

### MVP 范围

本次新增的 MVP 范围包括：

1. 月视图日历面板。
2. 查看每日任务。
3. 按日期创建任务。
4. 编辑任务。
5. 删除未完成任务。
6. 今日高亮。
7. 上月 / 下月切换。
8. 回到今天。
9. 移动端基础适配。

暂不纳入 MVP：

1. 周视图。
2. 日视图。
3. 拖拽任务改日期。
4. 重复任务。
5. 农历 / 节假日。
6. 多日任务。
7. 日历订阅。
8. 第三方日历同步。
9. 孩子自主创建任务。

### 交互说明

1. 点击空白日期：打开创建任务弹窗 / 抽屉。
2. 点击已有任务：打开任务详情。
3. 在任务详情中可以编辑或删除。
4. 删除操作必须二次确认。
5. 保存成功后日历立即刷新。
6. 创建任务默认日期为当前点击的日期。
7. 移动端优先使用底部抽屉或全屏表单，避免月视图日期格子过窄导致内容拥挤。

### 数据模型影响

日历面板优先复用现有任务数据，不新增独立日历表。

可复用字段：

| 现有字段 | 日历用途 |
|---|---|
| `study_task.family_id` | 家长查看当前家庭任务 |
| `study_task.child_user_id` | 区分所属孩子，孩子端只展示自己的任务 |
| `study_task.title` | 日历格子中的任务标题 |
| `study_task.description` | 任务说明和完成要求 |
| `study_task.due_date` | 日历日期 |
| `study_task.due_time` | 截止时间 |
| `study_task.reward_points` | 日历展示任务积分 |
| `study_task.need_photo` | 是否需要照片证明 |
| `study_task.status` | 日历任务状态 |
| `study_task.deleted_at` | 软删除后不再展示 |

建议补充字段：

| 字段 | 说明 | 阶段 |
|---|---|---|
| `voided_at` | 已完成任务被作废的时间 | 后续扩展 |
| `voided_by_user_id` | 作废操作人 | 后续扩展 |
| `void_reason` | 作废原因 | 后续扩展 |

如果后续允许处理已完成任务，应优先采用作废或归档，不应直接硬删除已发放积分的任务。

### 验收标准

1. 家长可以进入日历面板查看当前月份任务。
2. 家长可以切换上个月、下个月。
3. 家长可以点击今天按钮回到当前月份。
4. 家长可以点击某一天创建任务。
5. 创建后的任务正确显示在对应日期。
6. 家长可以点击任务进入详情。
7. 家长可以编辑任务信息。
8. 编辑后日历中的任务信息同步更新。
9. 家长可以删除未完成任务。
10. 删除后任务不再显示在日历中。
11. 孩子只能查看自己的任务，不能编辑或删除任务。
12. 日历中的任务状态与任务列表保持一致。
13. 积分发放仍然只发生在家长审核通过后。
14. 移动端可以正常查看当前月、切换月份、创建任务和查看任务详情。

### 待确认问题

1. 现有产品约束中“MVP 阶段不做多孩子复杂组织架构”，但日历需求要求家长查看所有孩子任务；MVP 可先支持当前家庭内已有孩子的任务聚合，是否需要同时补充家庭孩子列表选择能力待确认。
2. 现有任务状态包含 `pending`、`submitted`、`ai_checking`、`parent_review`、`confirmed`、`needs_resubmit`，日历需求中的“已驳回、已过期”需要作为展示态映射，还是扩展为持久化状态待确认。
3. 已完成任务作废会影响积分流水一致性；MVP 是否只删除未完成任务、把已完成作废放入后续扩展待确认。

## 8.9 薄弱点分析

### 功能目标

基于一段时间内的任务完成情况、错题记录、AI 检查结果，帮助家长发现孩子的薄弱科目、薄弱知识点和执行问题。

### 分析维度

1. 科目维度：数学、语文、英语等。
2. 任务类型维度：作业、预习、复习、错题、阅读、背诵等。
3. 完成情况维度：完成率、延期率、补充率。
4. 错题维度：错题数量、重复错题、知识点集中度。
5. AI 异常维度：图片无效、未完成、内容不匹配、模糊等。

### 输出内容

1. 本周薄弱科目。
2. 高频薄弱知识点。
3. 经常未完成的任务类型。
4. 需要家长重点关注的问题。
5. 下周建议练习方向。

### 验收标准

1. 系统可以基于一周数据生成薄弱点摘要。
2. 每个薄弱点需要有数据来源，如错题数量、任务异常次数。
3. 输出建议要具体，避免空泛。
4. 家长能看懂并用于安排下一周任务。

## 8.10 AI 每周学习报告

### 生成时间

### 报告结构

| 科目 | 任务数 | 完成率 | 错题数 | 风险提示 |
|---|---:|---:|---:|---|

| 语文 | 示例 | 示例 | 示例 | 示例 |
| 英语 | 示例 | 示例 | 示例 | 示例 |

#### 8.10.3 执行情况分析

1. 哪些天执行较好。

# 11. AI 能力设计

## 11.1 AI 能力分层

MVP 阶段 AI 能力分为四层：

1. 图片理解：识别上传图片中的作业内容、页码、题目、空白区域等。
2. 完成度判断：结合任务要求判断是否完成。
3. 错题与薄弱点提取：从上传内容、备注和家长标记中提取问题。
4. 报告生成：基于一周数据生成结构化周报。

## 11.2 AI 完成度检查 Prompt 设计方向

### 输入

1. 任务标题。
2. 任务说明。
3. 科目。
4. 任务类型。
5. 上传图片。
6. 孩子备注。

### 输出 JSON 示例

```json
{
  "is_related": true,
  "match_score": 0.86,
  "completion_level": "completed",
  "confidence": 0.78,
  "abnormal_flags": ["some_blank_questions"],
  "wrong_question_candidates": [
    {
      "description": "第 5 题疑似未完成或答案为空",
      "knowledge_point": "计算题",
      "confidence": 0.66
    }
  ],
  "summary": "上传内容与数学练习任务基本匹配，大部分题目已完成，但第 5 题附近存在空白，建议家长重点复查。",
  "suggestion_action": "parent_review"
}
```

## 11.3 AI 周报 Prompt 设计方向

### 输入

1. 一周任务统计。
2. 每日完成率。
3. 按科目统计。
4. AI 异常记录。
5. 错题记录。
6. 家长确认记录。

### 输出结构

1. 本周总体表现。
2. 执行情况分析。
3. 科目表现分析。
4. 薄弱点分析。
5. 下周行动建议。
6. 家长关注重点。

## 11.4 AI 可靠性原则

1. AI 不能假装确定，无法判断时必须输出“无法判断”。
2. AI 检查结果只作为辅助建议，不作为最终判定。
3. 涉及孩子学习评价时，语气要客观、温和，避免打击孩子。
4. AI 建议必须基于数据，不允许泛泛输出。
5. 家长必须能查看原始图片并覆盖 AI 判断。

# 13. 非功能性需求

## 13.1 性能要求

1. 今日任务列表加载时间不超过 2 秒。
2. 图片上传后 3 秒内展示本地或云端预览。
3. AI 检查可异步执行，不阻塞孩子提交。
4. AI 检查结果建议在 1 分钟内返回，复杂图片可延迟。
5. 日历面板当前月任务加载时间不超过 2 秒。

## 13.2 可用性要求

1. 孩子端操作必须简单，避免复杂配置。
2. 家长端重点突出“今日是否完成”和“哪些需要我处理”。
3. 错误提示必须明确，如图片上传失败、AI 检查失败。
4. 关键状态要明显，如“需补充”“待确认”。
5. 日历月视图在移动端需要避免任务文本互相遮挡，必要时只展示摘要并通过点击查看详情。

## 13.3 安全与隐私要求

1. 孩子数据只能被绑定家长查看。
2. 图片、错题、报告均属于家庭隐私数据。
3. 上传图片需要存储在受控云存储中。
4. 需要限制公开访问图片 URL。
5. AI 调用时避免传递不必要的个人身份信息。
6. 后续如正式上线，需要补充隐私政策和家长授权机制。

## 13.4 稳定性要求

1. AI 检查失败不能影响基础打卡流程。
2. 图片上传失败要支持重试。
3. 周报生成失败要支持重新生成。
4. 家长端必须始终可以查看原始上传内容。

4. 孩子查看今日任务。
5. 孩子拍照上传并提交。
6. 家长今日看板。
7. 家长确认或要求补充。

### 交付标准

1. 一个家庭可以正常使用。
2. 每天可以创建、完成、查看任务。
3. 任务状态流转完整。
4. 图片上传稳定可用。

## 14.2 第二阶段：积分愿望激励

### 目标

在打卡闭环中加入积分愿望激励，提升孩子持续完成任务的动力。

### 功能范围

1. 家长创建或编辑未完成任务时设置任务积分。
2. 家长确认任务完成后自动发放积分。
3. 孩子提交愿望。
4. 家长设置愿望所需积分或驳回。
5. 孩子发起兑换申请，家长确认兑换。
6. 积分获得和扣减记录流水。

### 交付标准

1. 积分发放和愿望兑换基础闭环可用。
2. 家长确认任务完成后才发放积分；后续接入 AI 后，AI 判断完成也不直接发放积分。
3. 同一任务积分只能发放一次。
4. 家长不能修改孩子提交的愿望原文。
5. 孩子只能发起兑换申请，家长确认后才扣减积分。

## 14.3 第三阶段：日历面板

### 目标

在现有任务系统上新增月视图日历面板，让家长和孩子可以按日期查看任务，并支持家长在日历中创建、编辑和删除未完成任务。

### 功能范围

1. 家长和孩子查看月视图日历面板。
2. 日历顶部显示当前年月。
3. 支持上个月 / 下个月切换。
4. 支持回到今天。
5. 今日高亮。
6. 每个日期格子展示当天任务摘要。
7. 家长按日期创建任务。
8. 家长点击任务进入详情并编辑任务。
9. 家长删除未完成任务。
10. 孩子只能查看自己的任务，不能创建、编辑或删除任务。
11. 移动端基础适配。

### 交付标准

1. 家长可以进入日历面板查看当前月份任务。
2. 家长可以切换上个月、下个月，并回到今天。
3. 家长可以点击某一天创建任务。
4. 创建后的任务正确显示在对应日期。
5. 家长可以点击任务进入详情并编辑任务。
6. 编辑后日历中的任务信息同步更新。
7. 家长可以删除未完成任务，删除后任务不再显示在日历和任务列表中。
8. 孩子只能查看自己的任务，不能编辑或删除任务。
9. 日历面板与任务列表展示同一套任务数据。
10. 积分发放仍然只发生在家长审核通过后。

## 14.4 第四阶段：AI 完成度检查、错题记录、薄弱点分析和周报

### 目标

在打卡闭环中加入 AI 检查能力，减少家长人工检查压力，并沉淀学习问题；基于任务、错题和 AI 检查数据生成薄弱点分析和每周学习报告。

### 功能范围

1. 上传后自动触发 AI 检查。
2. 识别图片内容是否匹配任务。
3. 判断完成度。
4. 输出异常提示。
5. 家长查看 AI 检查结论。
6. 家长反馈 AI 是否准确。
7. 家长手动添加错题。
8. AI 提取疑似错题。
9. 错题列表。
10. 按科目筛选错题。
11. 错题关联任务和图片。
12. 周任务统计。
13. 科目统计。
14. 错题统计。
15. 薄弱点分析。
16. AI 生成周报。
17. 历史周报查看。

### 交付标准

1. AI 检查结果可展示。
2. AI 无法判断时有明确提示。
3. 家长可覆盖 AI 结果。
4. AI 检查失败不影响基础流程。
5. AI 原始返回必须存入 `raw_result`。
6. 错题可添加、查看、标记掌握状态。
7. 错题能进入周报统计。
8. 每周可以生成一份结构化报告。
9. 报告内容与实际数据相关。
10. 下周建议具体可执行。

# 15. 推荐技术实现方案

## 15.1 客户端

MVP 阶段建议优先使用以下方式之一：

1. 微信小程序：适合家庭场景，家长和孩子使用门槛低。
2. H5 + 移动端适配：开发速度快，适合快速验证。
3. App 暂不建议第一版开发，成本较高。

推荐优先级：微信小程序 > H5 > App。

## 15.2 后端

建议采用轻量后端架构：

1. 用户与家庭关系服务，基于 MySQL 用户表实现用户名和密码登录。
2. 学习任务服务。
3. 打卡提交服务。
4. 图片上传服务。
5. 积分与愿望服务。
6. AI 检查任务服务。
7. 周报生成服务。

## 15.3 存储

1. 关系型数据库：MySQL。
2. 对象存储：用于保存上传图片。
3. Redis：用于异步任务状态、缓存和限流，可选。
4. 消息队列：用于 AI 检查异步任务，可选，MVP 可先用任务表轮询。

## 15.4 AI 模型调用

MVP 可以使用多模态大模型完成图片理解和文本生成。

主要调用场景：

1. 图片完成度检查。
2. 错题候选提取。
3. 周报生成。

建议将 AI 调用设计成独立服务，方便后续更换模型、优化 Prompt、控制成本。

# 18. 风险与应对

## 18.1 孩子不愿意打卡

### 风险

孩子觉得麻烦，导致打卡率低。

### 应对

1. 孩子端操作尽量少。
2. 默认只需勾选 + 拍照 + 提交。
3. 可以增加连续打卡反馈。
4. 家长布置任务不宜过多，每天控制在合理数量。

## 18.2 上传图片无效

### 风险

孩子上传模糊图片、无关图片或只上传封面。

### 应对

1. AI 检查图片是否清晰、是否相关。
2. 家长可以要求补充。
3. 提交页给出拍照示例。
4. 对无效上传进行统计。

## 18.3 AI 判断不准确

### 风险

AI 对手写题、复杂题型、模糊图片识别不稳定。

### 应对

1. AI 结果只作为辅助。
2. 输出置信度和无法判断状态。
3. 保留家长人工确认。
4. 收集家长反馈优化 Prompt。

## 18.4 家长坚持不下来

### 风险

家长前几天积极布置任务，后续停止使用。

### 应对

1. 支持任务模板。
2. 支持复制昨天任务。
3. 周报给出下周任务建议。
4. 后续支持一键生成任务计划。

## 18.5 周报空泛

### 风险

周报只是模板化总结，无法指导行动。

### 应对

1. 周报必须绑定真实数据。
2. 建议必须具体到科目、知识点和练习方式。
3. 报告中展示证据，如错题数量和异常任务。

## 18.6 积分激励偏离学习目标

### 风险

孩子只关注积分和兑换，忽视任务质量，或家长设置的积分过高、过低导致激励失衡。

### 应对

1. 积分只在家长确认任务完成后发放。
2. AI 结果和上传图片仍作为家长确认依据。
3. 家长创建任务时设置积分，避免系统自动放大奖励。
4. 愿望兑换必须由家长二次确认。
5. 第一版周报不展示积分和愿望数据，避免将激励数据误解为学习效果。

# 21. 结论

第一版 MVP 的关键不是做一个复杂的 AI 教育系统，而是验证家庭学习管理中的最小闭环是否成立：

```text
家长计划 → 孩子执行 → 拍照打卡 → AI 辅助检查 → 家长确认发放积分 → 孩子兑换心愿 → 每周复盘 → 下周调整
```

只要这个闭环跑通，并且家长确实感觉检查成本下降、孩子确实更清楚每天任务、积分愿望能带来正向反馈、周报确实能指导下一周安排，这个产品就具备继续迭代的基础。

第一版研发应优先保证任务打卡闭环稳定，再逐步增强 AI 检查、错题沉淀和周报分析能力。
