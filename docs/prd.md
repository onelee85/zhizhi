# 知知小助手第一版 MVP 产品需求文档

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | 知知小助手 |
| 文档类型 | 产品需求文档 PRD |
| 版本 | V1.0 / MVP 版本 |
| 当前定位 | 家庭学习打卡与积分激励工具 |
| 目标用户 | 自家单孩子家庭，由家长端和孩子端共同使用 |
| 核心目标 | 通过“家长创建任务 → 孩子完成打卡 → 家长确认 → 发放积分 → 孩子兑换心愿”的闭环，帮助孩子建立稳定的学习执行习惯，并让家庭激励过程清晰、可预期、可追溯 |

---

## 2. 产品背景

### 2.1 业务背景

孩子即将从小学升入民办初中，学习节奏、作业量和考试频率都会提升。家长希望孩子提前建立稳定的学习执行机制，同时能够持续掌握每天的任务完成情况。

当前家庭学习管理中存在以下问题：

1. 家长布置任务后，无法清楚追踪孩子是否完成。
2. 学习要求主要依赖口头沟通，容易遗漏或反复催促。
3. 孩子不知道每天具体要做什么，也缺少完成后的及时反馈。
4. 家庭奖励缺少统一规则，孩子难以预期努力与心愿之间的关系。
5. 已完成任务、照片、确认记录和积分变化缺少可追溯记录。

### 2.2 产品机会

当前产品聚焦三类价值：

1. **过程记录**：家长创建清晰任务，孩子完成后打卡并提交凭证。
2. **人工确认**：家长查看孩子提交的内容，确认通过或要求补充。
3. **正向激励**：任务通过后发放积分，孩子积累积分兑换经家长认可的心愿。

第一版 MVP 不接入 AI。产品首先验证基础任务与家庭激励闭环能否持续运行，再根据真实使用数据决定是否增加智能检查、错题分析和周报。

## 3. 产品定位

### 3.1 一句话定位

面向家长和孩子的家庭学习打卡与积分激励工具，帮助家长清晰安排任务，帮助孩子通过完成、反馈和心愿兑换建立学习执行习惯。

### 3.2 产品核心价值

| 用户 | 核心价值 |
|---|---|
| 家长 | 快速安排任务，集中处理待确认事项，清楚查看任务和积分记录 |
| 孩子 | 每天任务清晰，完成后及时得到反馈，能够看到积分和心愿进度 |
| 家庭 | 将口头催促变成“任务清单 + 打卡凭证 + 家长确认 + 积分心愿”的可追溯闭环 |

### 3.3 当前使用范围

1. MVP 仅面向自家真实使用。
2. 当前默认一个家庭、一个孩子，不建设多孩子管理体验。
3. 家长和孩子均使用应用内用户名和密码登录。
4. 当前以移动端使用为主，同时支持桌面端。
5. 连续真实使用 14 天后，再评估下一阶段范围。

### 3.4 MVP 不做什么

当前 MVP 暂不包含：

1. AI 图片检查或自动批改。
2. AI 任务状态、AI 异常统计和 AI 效果指标。
3. 错题自动识别、知识点分析和薄弱点诊断。
4. AI 周报和学习计划推荐。
5. 完整在线教学、课程、直播课、录播课或班级管理。
6. 多孩子、多家庭、多老师等复杂组织能力。
7. 负积分、积分过期或惩罚性扣分。
8. 重复任务、第三方日历同步和复杂消息推送。

Alibaba Bailian 保留为后续智能能力的技术选项，当前版本不接入、不依赖，也不对用户承诺 AI 功能。

## 4. 目标用户与角色

### 4.1 家长

#### 用户特征

1. 希望孩子建立每日学习习惯。
2. 希望任务要求能够明确传达，减少反复提醒。
3. 希望集中查看孩子提交的结果并及时反馈。
4. 希望通过一致的积分规则进行正向激励。

#### 核心诉求

1. 快速创建和管理任务。
2. 优先看到待自己处理的任务。
3. 查看孩子上传的凭证和备注。
4. 确认通过或明确要求补充。
5. 管理积分、愿望和兑换记录。

### 4.2 孩子

#### 用户特征

1. 小学高年级或初中预备阶段。
2. 自律能力尚未完全建立。
3. 对复杂、繁琐的操作敏感。
4. 需要清晰任务和及时正向反馈。

#### 核心诉求

1. 每天知道需要完成什么。
2. 用尽可能少的步骤完成打卡。
3. 清楚知道任务为什么需要补充。
4. 查看积分来源、当前余额和心愿进度。
5. 达到目标后可靠地完成心愿兑换。

## 5. 产品目标与指标

### 5.1 MVP 核心假设

1. 家长愿意持续创建和确认学习任务。
2. 孩子愿意查看任务并主动完成打卡。
3. 照片或备注可以作为有效的完成凭证。
4. 过程型积分能够提升孩子持续完成任务的积极性。
5. 清晰的心愿进度和可靠兑现能够增强激励效果。
6. 日历和历史记录能够帮助家庭规划与回顾任务。

### 5.2 激励原则

1. 积分以奖励执行过程为主，包括按要求完成、主动反馈和认真补充。
2. 任务质量是家长确认的最低门槛，不按正确率进行精细计分。
3. 家长要求补充时暂不发分；孩子补充后通过，发放原任务积分。
4. 不设置负积分、积分过期或因任务失败扣除已有积分。
5. 家长确认完成是任务积分发放的唯一触发点。
6. 同一任务只能发放一次积分。
7. 心愿经家长通过并设置积分后，所需积分不得随意提高。
8. 心愿兑换应被视为家庭承诺；兑换申请后的拒绝仅用于客观上无法兑现的情况，并必须退款。

### 5.3 核心业务指标

| 指标 | 说明 | MVP 目标 |
|---|---|---|
| 周有效激励闭环率 | 到期任务中，孩子完成提交且家长在 24 小时内确认并发放积分的比例 | ≥ 70% |
| 家长任务创建频率 | 连续 7 天内创建任务的天数 | ≥ 5 天 |
| 孩子打卡完成率 | 孩子完成并提交的任务比例 | ≥ 70% |
| 上传有效率 | 家长能够据此判断是否完成的提交比例 | ≥ 80% |
| 家长 24 小时确认率 | 提交后 24 小时内被家长处理的任务比例 | ≥ 80% |
| 补充完成率 | 被要求补充后最终重新提交并通过的比例 | ≥ 70% |
| 家长第二周复用率 | 第 2 周仍持续创建和确认任务 | 自家连续使用 |
| 心愿激励完成度 | 完成愿望提交、设置积分、兑换申请和最终兑现 | 14 天内至少完成 1 次 |
| 积分公平性评分 | 家长和孩子对规则是否清楚、公平的主观评分 | 双方均 ≥ 4/5 |

## 6. 核心使用流程

### 6.1 家长侧主流程

1. 家长登录系统。
2. 创建学习任务并设置任务积分。
3. 任务发布到孩子任务清单和日历。
4. 孩子提交后，任务直接进入待家长确认。
5. 家长查看照片、孩子备注和任务要求。
6. 家长选择确认通过或要求补充，并可填写简短备注。
7. 确认通过后，系统发放任务积分并记录流水。
8. 家长查看孩子提交的愿望，设置所需积分并通过，或说明原因后驳回。
9. 孩子申请兑换后，家长确认已经兑现；客观上无法兑现时说明原因并退款。

### 6.2 孩子侧主流程

1. 孩子登录系统。
2. 查看今日、逾期或已完成任务。
3. 完成任务并进入打卡页。
4. 根据任务要求上传照片，可填写备注。
5. 提交后等待家长确认。
6. 如被要求补充，查看家长原因并重新上传。
7. 家长确认后获得对应积分。
8. 查看积分余额、积分流水和心愿进度。
9. 提交自己的心愿。
10. 积分足够时发起兑换申请。
11. 查看兑换确认或退款结果。

### 6.3 任务确认流程

1. 孩子提交任务。
2. 系统将任务状态更新为“待家长确认”。
3. 家长查看任务要求、图片和备注。
4. 家长确认通过，系统发放积分；或要求补充，填写补充原因。
5. 孩子补充提交后，任务再次进入“待家长确认”。

### 6.4 愿望与积分兑换流程

1. 家长创建任务时设置可获得的积分。
2. 孩子完成任务并提交。
3. 家长确认完成，系统发放积分并记录流水。
4. 孩子提交愿望名称和说明。
5. 家长设置所需积分并通过，或填写原因后驳回。
6. 愿望通过后，孩子可查看已有积分、所需积分、还差积分和完成进度。
7. 积分足够时，孩子发起兑换申请。
8. 系统原子校验余额，立即扣减积分并记录流水。
9. 家长确认已经兑现后，愿望标记为已兑换，不重复扣分。
10. 客观上无法兑现时，家长填写原因并拒绝兑换；系统原额退款，愿望恢复为可兑换。

## 7. 功能范围与优先级

### 7.1 第一版最关键的 7 个功能

1. 家长创建、编辑和删除未完成任务。
2. 孩子查看任务清单。
3. 孩子上传凭证并提交打卡。
4. 家长确认通过或要求补充。
5. 积分发放、流水记录和心愿兑换。
6. 日历面板按日期查看和管理任务。
7. 历史任务自动归档与只读查看。

### 7.2 优先级

| 优先级 | 模块 | 说明 |
|---|---|---|
| P0 | 任务打卡与人工确认 | 当前 MVP 成立的基础 |
| P0 | 积分与心愿兑换 | 当前阶段要验证的核心价值 |
| P0 | 积分与兑换流水 | 保证家庭奖励清楚、可追溯 |
| P1 | 日历面板 | 支持按日期规划和查看任务 |
| P1 | 历史任务归档 | 避免普通列表堆积，同时保留记录 |
| P2 | 任务模板和复制昨日任务 | 降低家长长期使用成本，后续迭代 |
| Future | AI、错题和智能周报 | 真实使用验证后再评估 |

# 8. 功能需求

## 8.1 家长创建和管理任务

### 功能目标

帮助家长创建清晰、可执行、可确认的学习任务。

### 主要字段

| 字段 | 说明 | 是否必填 |
|---|---|---|
| 任务标题 | 如“完成数学计算练习第 3 页” | 是 |
| 科目 | 语文、数学、英语、其他 | 是 |
| 任务类型 | 作业、预习、复习、错题、阅读、背诵、练习 | 是 |
| 任务说明 | 页码、题号、完成标准等 | 是 |
| 任务日期 | 任务所属日期 | 是 |
| 截止时间 | 当天几点前完成 | 否 |
| 任务积分 | 非负整数，默认 1 分，可设置为 0 | 是 |
| 是否需要拍照 | 默认需要 | 是 |
| 备注 | 给孩子的补充说明 | 否 |

### 业务规则

1. 当前家庭只有一个孩子，不要求家长每次选择孩子。
2. 新建任务状态为“待完成”。
3. 仅“待完成”和“需补充”任务可编辑。
4. 编辑时可修改标题、说明、日期、积分、截止时间和照片要求。
5. 已提交、待家长确认和已确认任务不可编辑。
6. 仅未提交的“待完成”任务可以删除。
7. 已确认任务通过历史归档保留，不支持删除或修改积分。

### 验收标准

1. 家长可以在 1 分钟内创建一条任务。
2. 任务保存后孩子端可立即查看。
3. 创建时必须填写标题、科目、类型、说明、日期和积分。
4. 任务积分只允许非负整数。
5. 编辑和删除权限符合任务状态限制。

## 8.2 孩子查看任务

### 页面内容

1. 今日、逾期、已完成三个视图。
2. 每项任务展示科目、标题、说明、截止时间、积分和状态。
3. 状态包括：待完成、待家长确认、已确认、需补充。
4. 已归档任务通过独立历史任务入口查看。
5. 点击任务可进入任务详情、打卡页或结果页。

### 验收标准

1. 孩子能够看到分配给自己的任务。
2. 已完成与未完成任务有明显区分。
3. 需补充任务明确展示家长原因和补充入口。
4. 从任务列表进入打卡不超过 3 步。

## 8.3 孩子打卡提交

### 功能说明

孩子完成任务后，可以勾选完成并按任务要求上传照片。照片可为作业本、练习册、试卷、阅读笔记、背诵记录等。

### 主要字段

| 字段 | 说明 |
|---|---|
| 任务 ID | 当前打卡对应任务 |
| 完成状态 | 孩子主动勾选完成 |
| 上传图片 | 需要照片时上传 1 到 9 张 |
| 孩子备注 | 可选，如“第 5 题不会” |
| 提交时间 | 系统自动记录 |

### 业务规则

1. 任务要求照片时，至少上传 1 张后才能提交。
2. 不要求照片的任务可以仅勾选完成并填写可选备注。
3. 单个任务最多上传 9 张图片。
4. 图片上传失败时允许重试，不提交不完整数据。
5. 提交后直接进入“待家长确认”。
6. 已提交任务不可重复提交。
7. “需补充”任务允许再次上传并生成新的提交记录。

### 验收标准

1. 孩子能够完成有照片和无照片两类任务的提交。
2. 家长能够查看所有提交图片和备注。
3. 上传失败有明确提示并支持重试。
4. 提交后任务直接进入待家长确认。
5. 补充提交后再次进入待家长确认。

## 8.4 家长完成看板与确认

### 页面内容

1. 当前任务总数。
2. 待完成数量。
3. 待家长确认数量。
4. 需补充数量。
5. 已确认数量。
6. 每个任务的状态、积分、图片和孩子备注。
7. 孩子当前可用积分入口。

### 排序规则

1. 待家长确认。
2. 需补充。
3. 今日和逾期待完成。
4. 未来任务。
5. 最近确认完成且尚未归档的任务。

### 家长操作

1. 确认通过。
2. 要求补充。
3. 填写简短备注或补充原因。

### 验收标准

1. 家长进入看板后优先看到待自己处理的任务。
2. 家长可以查看任务要求、提交图片和孩子备注。
3. 要求补充时必须向孩子展示明确原因。
4. 确认通过后立即发放积分。
5. 同一任务重复调用确认操作不会重复发分。
6. 页面处理后状态和积分实时刷新。

## 8.5 积分与愿望兑换

### 功能目标

建立清楚、可预期的家庭正向激励机制，使孩子能够理解完成任务与实现心愿之间的关系。

### 任务积分规则

1. 家长在创建或编辑未完成任务时设置积分。
2. 积分为非负整数，默认 1 分，可设置为 0。
3. 孩子提交或任务进入待家长确认时不发分。
4. 家长确认任务完成后才发放积分。
5. 同一任务只能发放一次积分。
6. 需补充任务暂不发分；补充通过后发放原任务积分。
7. 不设置负积分、积分过期或任务失败扣分。
8. 所有积分获得、扣减和返还均记录流水。

### 愿望规则

1. 孩子提交愿望标题和可选说明。
2. 愿望提交后进入“待家长设置积分”。
3. 家长只能设置所需积分并通过，或填写原因后驳回，不能修改孩子原文。
4. 愿望通过后，所需积分不得提高。
5. 被驳回愿望可由孩子修改后重新提交，或直接删除。
6. 孩子可查看当前积分、所需积分、还差积分和完成进度。
7. 积分不足时不能发起兑换申请。
8. 积分足够时，申请兑换会立即预扣积分并记录流水。
9. 多个待确认兑换申请不得透支余额。
10. 家长确认兑换只标记“已兑换”，不重复扣分。
11. 家长仅在客观上无法兑现时拒绝兑换，必须填写原因。
12. 拒绝兑换后原额退款，愿望恢复为“可兑换”，并记录退款流水。

### 愿望状态

| 状态 | 说明 |
|---|---|
| 待家长设置积分 | 孩子已提交，等待家长处理 |
| 可兑换 | 家长已设置积分，孩子可继续积累或申请兑换 |
| 已驳回 | 家长说明原因后驳回 |
| 兑换申请中 | 孩子已申请，积分已预扣 |
| 已兑换 | 家长确认心愿已兑现 |

### 验收标准

1. 家长确认任务后积分只发放一次。
2. 补充任务通过后正常获得原任务积分。
3. 孩子能够查看余额、流水和愿望进度。
4. 愿望批准后所需积分不能随意提高。
5. 兑换申请会原子扣分，不能造成余额透支。
6. 确认兑换不重复扣分。
7. 拒绝兑换必须填写原因，并自动全额退款。
8. 每次获得、扣减和退款均可追溯。

## 8.6 日历面板

### 功能目标

通过月视图按日期聚合任务，让家长和孩子直观看到每天的任务安排。

### 家长权限

1. 查看当前孩子的月任务。
2. 点击日期创建任务，并默认使用该日期。
3. 点击任务进入详情。
4. 编辑“待完成”和“需补充”任务。
5. 删除未提交的“待完成”任务。

### 孩子权限

1. 只读查看分配给自己的任务。
2. 点击任务进入详情、打卡页或结果页。
3. 未来任务可查看，但不能提前提交。
4. 不允许创建、编辑或删除任务。

### 视图要求

1. 默认展示月视图。
2. 支持上个月、下个月和回到今天。
3. 今天有明显标识。
4. 桌面端日期格展示任务摘要。
5. 移动端日期格展示状态点和任务数，完整信息放在选中日期面板。
6. 已归档任务仍在对应日期展示，并标记“已归档”。

### 日历展示状态

| 展示状态 | 对应任务状态 |
|---|---|
| 待完成 | `pending` |
| 待家长确认 | `parent_review` |
| 已完成 | `confirmed` |
| 需补充 | `needs_resubmit` |
| 已过期 | 截止时间已过且仍为 `pending` 或 `needs_resubmit` 的派生展示状态 |

“已过期”仅为展示状态，不新增持久化任务状态。

### 验收标准

1. 家长和孩子能够查看当前月份任务。
2. 月份切换和回到今天正常工作。
3. 家长可以按日期创建任务。
4. 编辑后任务列表和日历同步更新。
5. 删除后任务列表和日历同步移除。
6. 孩子只能查看自己的任务。
7. 未来任务不能提前提交。
8. 移动端无横向溢出，任务信息可通过选中日期查看。

## 8.7 历史任务归档

### 归档规则

1. 仅归档状态为 `confirmed` 的任务。
2. 以家长确认通过时间为起点，超过 7 天后自动进入历史任务。
3. 归档是查询和展示规则，不改变任务状态。
4. 归档不影响已发积分，不产生新积分流水。
5. MVP 不支持手动归档、取消归档或批量归档。

### 展示规则

1. 家长和孩子的普通任务列表默认不展示已归档任务。
2. 家长端和孩子端均提供历史任务入口。
3. 历史任务只读，不支持编辑、删除、重新提交或重新确认。
4. 日历面板继续展示已归档任务。
5. 历史详情保留任务、提交照片、孩子备注、家长确认记录和积分信息。

### 验收标准

1. 确认未超过 7 天的任务仍显示在普通列表。
2. 确认超过 7 天的任务只在历史入口和日历中展示。
3. 家长只能查看当前家庭的历史任务。
4. 孩子只能查看自己的历史任务。
5. 历史任务详情完整且只读。

# 10. 数据与状态设计

## 10.1 当前核心数据实体

1. `user`：家长和孩子应用用户。
2. `family_member`：家庭关系和数据隔离。
3. `study_task`：任务信息、日期、积分和状态。

6. `parent_review`：家长确认或要求补充的记录。
7. `child_point_account`：孩子当前积分余额。
8. `point_ledger`：积分获得、兑换扣减和退款流水。
9. `wish`：孩子愿望和兑换状态。

当前 MVP 不要求 `ai_check_result`、`wrong_question` 或 `weekly_report` 数据表。后续启动对应版本时再补充设计。

## 10.2 学习任务核心字段

| 字段 | 说明 |
|---|---|
| `family_id` | 所属家庭 |
| `child_user_id` | 所属孩子 |
| `creator_user_id` | 创建家长 |
| `subject` | 科目 |
| `task_type` | 任务类型 |
| `title` | 标题 |
| `description` | 任务说明 |
| `due_date` | 任务日期 |
| `due_time` | 截止时间 |
| `reward_points` | 家长设置的任务积分 |
| `need_photo` | 是否需要照片 |
| `status` | 当前任务状态 |
| `deleted_at` | 未提交任务软删除时间 |

`need_ai_check` 不属于当前产品需求。代码或数据库中的兼容字段不代表当前版本提供 AI 能力。

## 10.3 当前任务状态

| 状态 | 说明 |
|---|---|
| `pending` | 待完成，孩子尚未提交 |
| `parent_review` | 待家长确认，孩子已提交或补充提交 |
| `confirmed` | 家长已确认完成，积分已发放或任务积分为 0 |
| `needs_resubmit` | 家长要求补充，等待孩子重新提交 |

`submitted` 和 `ai_checking` 不属于当前产品状态设计。代码中的历史兼容状态应在后续实施中迁移或兼容映射为 `parent_review`。

## 10.4 家长确认记录

| 字段 | 说明 |
|---|---|
| `task_id` | 关联任务 |
| `submission_id` | 关联本次提交 |
| `parent_user_id` | 操作家长 |
| `review_result` | `pass` 或 `need_resubmit` |
| `comment` | 家长备注或补充原因 |
| `reviewed_at` | 操作时间 |

## 10.5 积分流水约束

1. 同一任务只能产生一条 `task_reward` 流水。
2. 同一次兑换请求最多产生一条 `wish_redeem` 扣款流水。
3. 兑换被拒绝时最多产生一条对应的 `wish_refund` 流水。
4. 所有积分操作必须在事务内校验家庭、孩子、余额和来源。
5. 当前余额不得小于 0。

## 10.6 愿望约束

1. 愿望标题和说明由孩子填写，家长不能修改。
2. 所需积分由家长设置。
3. 愿望通过后，所需积分不可提高。
4. 兑换申请必须使用独立请求 ID。
5. 兑换申请和退款流水必须关联同一请求 ID。

# 11. 状态流转

## 11.1 任务状态流转

```text
待完成 pending
  ↓ 孩子提交
待家长确认 parent_review
  ↓                     ↘
家长确认完成              家长要求补充
confirmed                needs_resubmit
                          ↓ 孩子补充提交
                        parent_review
```

## 11.2 积分发放流转

```text
孩子提交任务
  ↓
待家长确认
  ↓
家长确认完成
  ↓
原子发放任务积分
  ↓
记录积分流水
```

约束：

1. 孩子提交不发分。
2. 待家长确认不发分。
3. 需补充不扣分，也不发分。
4. 家长确认完成是唯一发分入口。
5. 同一任务确认只能发放一次积分。

## 11.3 愿望状态流转

```text
待家长设置积分
  ↓                 ↘
可兑换               已驳回
  ↓ 孩子申请兑换
兑换申请中（立即预扣积分）
  ↓                         ↘
家长确认已兑现                家长拒绝并填写原因
  ↓                         ↓
已兑换                       全额退款并回到可兑换
```

# 12. 非功能性需求

## 12.1 性能

1. 任务列表和当前月日历加载时间不超过 2 秒。
2. 图片上传后 3 秒内展示预览。
3. 任务确认和积分变化应在操作完成后立即刷新。

## 12.2 可用性

1. 孩子端打卡流程尽量控制在三步内。
2. 家长端优先突出“待我确认”和“需补充”。
3. 图片上传、网络错误和余额不足必须有明确提示。
4. 关键状态使用一致文案和视觉标识。
5. 移动端不得出现横向滚动。

## 12.3 安全与隐私

1. 孩子数据只能由当前家庭绑定家长查看。
2. 任务、图片、积分和愿望均属于家庭隐私数据。
3. 所有写入必须验证登录用户、角色、家庭和孩子归属。
4. 图片访问必须经过受控服务，不暴露服务器真实文件路径。
5. 当前自家使用可采用本地受控存储；公开内测前需评估对象存储、访问签名、备份和删除策略。
6. 正式上线前需要隐私政策和家长授权机制。

## 12.4 稳定性

1. 图片上传失败支持重试。
2. 读取请求遇到临时网络故障可以自动重试。
3. 写入请求不得因客户端自动重试造成重复任务、重复提交或重复积分流水。
4. 积分发放、兑换预扣和退款必须幂等。
5. 家长始终可以查看原始提交内容。

# 13. MVP 开发阶段

## 13.1 第一阶段：任务打卡与人工确认

### 目标

跑通家长创建任务、孩子查看和提交、家长确认或要求补充的闭环。

### 交付标准

1. 自家家长和孩子账号可以正常登录。
2. 每天可以创建、查看、提交和确认任务。
3. 孩子提交后直接进入 `parent_review`。
4. 任务状态流转完整。
5. 图片上传稳定可用。

## 13.2 第二阶段：积分与心愿激励

### 目标

验证过程型积分和心愿兑换是否能提升孩子持续完成任务的积极性。

### 交付标准

1. 家长确认后才发放积分。
2. 同一任务积分只发放一次。
3. 孩子能够查看余额、流水和心愿进度。
4. 兑换申请原子预扣积分，余额不能透支。
5. 家长确认兑换不重复扣分。
6. 拒绝兑换必须填写原因并原额退款。

## 13.3 第三阶段：日历与历史任务

### 目标

支持按日期规划任务，并避免普通任务列表长期堆积已完成记录。

### 交付标准

1. 家长和孩子能够查看月视图。
2. 家长可以按日期创建、编辑和删除符合条件的任务。
3. 孩子只能查看自己的任务。
4. 确认超过 7 天的任务进入历史任务。
5. 历史任务只读，日历仍保留归档展示。

## 13.4 第四阶段：真实使用验证

### 目标

连续使用 14 天，验证家庭激励闭环，而不是继续增加功能。

### 重点观察

1. 家长创建任务和确认任务的持续性。
2. 孩子主动查看和完成任务的比例。
3. 补充原因是否清楚、补充后是否完成。
4. 积分规则是否被双方认为公平。
5. 心愿进度是否能够持续产生激励。
6. 是否至少完成一次可靠的心愿兑现。

# 17. 风险与应对

## 17.1 孩子不愿意打卡

### 应对

1. 打卡操作保持简单。
2. 不需要照片的任务允许无照片提交。
3. 任务数量保持合理。
4. 使用清晰的积分和心愿进度提供正向反馈。

## 17.2 家长处理不及时

### 应对

1. 看板优先展示待家长确认。
2. 统计 24 小时确认率。
3. 后续可增加应用内待处理提醒。
4. 避免要求家长填写冗长评价。

## 17.3 上传凭证无效

### 应对

1. 提交页展示简短拍照提示。
2. 家长可以要求补充并说明原因。
3. 统计无效提交和补充完成情况。

## 17.4 积分激励偏离学习目标

### 应对

1. 积分奖励执行过程，质量只作为最低门槛。
2. 不按成绩或正确率进行复杂计分。
3. 不使用负积分或惩罚性扣分。
4. 家长和孩子定期讨论积分是否公平。
5. 愿望目标保持合理、可实现。

## 17.5 心愿承诺无法兑现

### 应对

1. 家长通过愿望前确认其可兑现性。
2. 愿望通过后不得随意提高积分门槛。
3. 拒绝兑换必须填写客观原因并退款。
4. 保留兑换和退款流水，避免家庭争议。

## 17.6 家长长期创建任务成本过高

### 应对

1. 先通过 14 天真实使用确认高频任务模式。
2. 后续增加常用任务模板和复制昨日任务。
3. 不在验证前引入复杂重复规则。

2. 按时完成率。
3. 补充次数和补充完成率。
4. 按科目和任务类型统计。
5. 规则生成的周总结，不使用 AI。

## 18.3 V1.3：AI 辅助检查

启用前提：14 天真实使用证明家长人工确认成本是主要问题，并完成准确率、隐私和成本评估。

计划能力：

1. 使用 Alibaba Bailian 或可替换的多模态服务。
2. 判断图片是否清晰、完整并与任务相关。
3. 输出完成、部分完成或无法判断。
4. 提供建议通过、建议复查或建议补充。
5. AI 失败时直接回退到人工确认。
6. 家长始终拥有最终判断权。
7. AI 不直接发积分、不扣分、不评价孩子能力。

后续 AI 状态独立于任务状态设计，不恢复 `ai_checking` 作为任务主状态。

## 18.4 V1.4：错题与薄弱点

启用前提：AI 图片检查达到可接受准确率，且家庭确实需要沉淀错题。

计划能力：

1. 家长手动添加错题。
2. AI 生成疑似错题候选，由家长确认。
3. 错题关联原任务和图片。
4. 错题掌握状态和二刷任务。
5. 基于真实错题数据生成薄弱点摘要。

## 18.5 V1.5：AI 周报

启用前提：已经积累稳定的任务、确认和错题数据。

计划能力：

1. 汇总任务完成、补充、错题和科目数据。
2. 生成有证据来源的学习复盘。
3. 给出具体、可执行的下周建议。
4. 支持历史周报。
5. 家长可以基于建议创建任务草稿。

## 18.6 V2.0：扩展家庭与学习能力

1. 多孩子管理。
2. 更完整的知识点体系。
3. 日历周视图、重复任务和拖拽调整日期。
4. 学习路径与阶段性计划。
5. 公开内测所需的账户、隐私、存储和运营能力。

# 20. 结论

第一版 MVP 验证的最小闭环是：

```text
家长创建任务
  → 孩子完成并打卡
  → 家长确认或要求补充
  → 确认后发放积分
  → 孩子查看心愿进度并申请兑换
  → 家长可靠兑现
```

当前产品的关键不是自动判断孩子学得怎么样，而是让任务、反馈和家庭奖励形成清晰稳定的日常机制。

AI、错题、薄弱点和智能周报统一放在后续版本。只有在基础闭环连续使用 14 天且真实数据证明有必要时，才进入下一阶段。
